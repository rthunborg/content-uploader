- source_spec: `_bmad-output/implementation-artifacts/spec-1-3-fleet-deck-design-tokens-themed-component-base.md`
  summary: The `dialogDestructive` button variant pairs white (`--surface` `#ffffff`) text on Core Red (`--destructive` `#e41f1f`), which computes to ~4.48:1 contrast — just below the 4.5:1 WCAG AA floor for its 16 px semibold (normal-weight) label — and neither approved palette foreground (white nor Black `#1a1a1a`, ~3.89:1) clears AA on Core Red; deferrable because the variant is dialog-only per the intent and is not yet mounted on any surface, so no current user is affected.
  evidence: `src/components/ui/button.tsx` defines `dialogDestructive: "bg-destructive text-surface …"` (white on `#e41f1f`). Recomputing the WCAG contrast: L(white)=1.0, L(#e41f1f)≈0.1845, ratio=(1.0+0.05)/(0.1845+0.05)≈4.48, under 4.5 for normal text (16 px `font-semibold`, weight 600, is not WCAG "large text": needs ≥18.66 px bold or ≥24 px). The intent's Block-If ("approved palette pairings cannot meet WCAG AA without changing a locked token") is implicated, but the intent also scopes destructive presentation to confirmation dialogs, none of which exist yet — so the pairing is not observable by any user until such a dialog is built. Resolution (large-text sizing for the confirm label, or a locked-token decision on Core Red) belongs to the story that first mounts a destructive confirmation dialog.

- source_spec: `_bmad-output/implementation-artifacts/spec-1-4-magic-link-authentication-the-single-front-door.md`
  summary: The `/auth/login` magic-link request discloses whether an account exists — an existing user yields the "check your inbox" success copy (`AUTH_COPY.linkSent`) while `signInWithOtp` returning an error (e.g. an unknown email under `shouldCreateUser: false`) yields the distinct `AUTH_COPY.requestFailed` copy — contrary to the intent's enumeration-safety criterion.

Collapsing both outcomes to a single generic confirmation message (and covering the unknown-email branch in tests) would close it.

- source_spec: `_bmad-output/implementation-artifacts/spec-1-4-magic-link-authentication-the-single-front-door.md`
  summary: The invite email template assumes the provider's `.RedirectTo` always carries a query string; an administrator invitation issued with a query-less `redirectTo` produces a malformed link (`http://host&token_hash=...&type=invite`) whose token collapses into the path and never reaches `/auth/confirm`.
  evidence: `supabase/templates/invite.html` builds the href as `{{ .RedirectTo }}&amp;token_hash={{ .TokenHash }}&amp;type=invite`, valid only when `.RedirectTo` already contains a `?`. The magic-link path is safe because `buildConfirmationUrl` always emits `?next=`, but invite `redirectTo` is supplied by the (future) admin caller; the only current caller is the e2e test, which happens to pass `?next=%2F`. Invitation onboarding is owned by later stories, so no production caller is reachable today, but the template will silently break the invite journey for any query-less `redirectTo`. A delimiter-robust template (or a guarantee that invite `redirectTo` always carries a query) plus a rendered-href assertion would resolve it.

- source_spec: `_bmad-output/implementation-artifacts/spec-1-4-magic-link-authentication-the-single-front-door.md`
  summary: The Story 1.4 headline guarantees that only the Playwright e2e suite exercises — real single-use enforcement, 15-minute expiry, and the accessible recovery CTA (44 px target + visible focus outline) — have no automated/CI execution path, so they can regress undetected under the default `npm test`.
  evidence: `vitest.config.ts` excludes `e2e/**`, and both `playwright.config.ts` and `e2e/fixtures/auth.ts` hard-require a manually started local Supabase (`npx supabase status`, throwing otherwise). Wiring `playwright test` (with `supabase start`) into the project's verification gate — or documenting it as a mandatory pre-merge step — would close the gap. This is a project-level CI gap not caused by Story 1.4 (the story has no mandate to establish CI).
